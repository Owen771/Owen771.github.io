01 - Ticketmaster

Created2024-07-28|Updated2024-07-28
|Post Views:

1. Requirement

non functional:

  • low latency search

out of scope:

  • event ranking or recommendation

2. Core entities + API

NOTE:

  • when book ticket, ususally there’s 2 phrase, one is reserve, one is purchase within a 10 min windows
  • if purchase windows passed, release ticket from reserve

JWT store user info for safety reason (otherwise ppl can post reserver on behalf of somebody else)

  • POST /booking/reserve: reserve ticket for 10 min
  • POST or PUT /booking/confirm: confirm the payment go thru = ticket sold

3. High level design

most simple version

impl 10 min windows

  1. acceptable: cron job
    ticket table added
  • status: available, reserved, booked
  • userId

cons: a ticket has chance that 19mins not available

  • 11:50 order, 12:00 expire
  • cron job ran at 11:59 -> next one is 12:09
  • 11:50 to 12:09 ticket is not available
  1. better: distributed lock by Redis

4 Deep Dive

  • add elasticsearch
  • CDC from postgres to elasticsearch
  • add CDN
  • add virtual waiting queue
  • add another for event,venue,performer, as they pretty much never change
