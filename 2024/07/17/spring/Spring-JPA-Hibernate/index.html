Part 1: The basic

App <-> JPA (Hibernate) <-> JDBC <-> DB

JPA (Java/Jakarta Persistence API)

  • Standard API for ORM
  • It’s only a spec (interface) and require an implmentation
    • Hibernate is an implementation of JPA

ORM (Object-relational mapping)

  • JPA maps Java objects to database tables, allowing developers to work with Java objects instead of writing SQL statements.

Create a new MySQL user: (springstudent, springstudent)

Create Student Table

Automatic Data Source Configuration

  • In Spring Boot, Hibernate is the default implementation of JPA and EntityManager is from JPA

  • EntityManager is main component for creating queries etc…

  • Based on configs, Spring Boot will automatically create the beans:

    • DataSource, EntityManager…
    • You can then inject these into your app, for example your DAO

Entity Class

Must have

  1. Annotated with @Entity
  2. A public or protected no-argument constructor
    • The class can have other constructors

  • @Id mark field as PK

  • @Table mark class with a DB table name

    • If not specified, database table name is same as the class

  • @Column is optional

    • If not specified, the column name is the same name as Java field, but not recommend this approach

  • PK cannot be null

DAO (Data Access Object)

  • Define method that perform ops to DB, e.g void save(Student student)

JPA Entity Manager

  • it interact with DB
  • the main component for saving/retrieving entities
  • DAO needs a JPA Entity Manager
  • it needs a Data Source, which defines database connection info

JPA Entity Manager and Data Source are automatically created by Spring Boot
• Based on the file: application.properties (JDBC URL, user id, password, etc …)
• We can autowire/inject the JPA Entity Manager into our Student DAO

JpaRepository vs EntityManager

Which one to use and when?

  • JpaRepository provides JPA database access with minimal coding

Short ans for now

  • If you need low-level control and flexibility, use EntityManager
  • If you want high-level of abstraction, use JpaRepository
  • Spring provide @Transactional
  • all non read-only ops need @Transactional, otherwise will get
    org.springframework.dao.InvalidDataAccessApiUsageException: No EntityManager with actual transaction available for current thread - cannot reliably process 'persist' call

@Repository

  • child of @Component
  • use for annotate DAO (seems like annotated in DAO impl instead of the DAO interface)

Translate JDBC exceptions

  • Spring provided translation of any JDBC exceptions
    • meaning if you have any checked JDBC exceptions, then Spring will translate these to unchecked exception (Why?? what for)

Checked exceptions are exceptions that are checked by the compiler at compile-time, either handles the exception (try-catch) or declares it in the method signature

Unchecked exceptions are exceptions that are not checked by the compiler at compile-time, that’s a programming error or an unexpected condition that cannot be recovered from.

  • e.g NullPointerException, IllegalArgumentException, and ArrayIndexOutOfBoundsException.

JPQL

  • All JPQL is based on entity name and field (the name that declare in Java Class and Object)

Create Database Tables from Java Code

  • Java -> JPA(Hibernate) -> SQL -> DB
  • it’s based on @Table and @Column to create

spring.jpa.hibernate.ddl-auto=<VALUE>, value:

  1. none: No action will be performed
  2. create-only: Database tables are only created
  3. drop: Database tables are dropped
  4. create: Database tables are dropped followed by database tables creation
  5. create-drop: Database tables are dropped followed by database tables creation. On application shutdown, drop the database tables (drop table after app shutdown, primarily useful for unit testing)
  6. validate: Validate the database tables schema
  7. update: Update the database tables schema (like if you add a new field, also will keep existing data)

Show SQL statement and the value sending into SQL

  • add config application.yaml

spring.jpa.hibernate.ddl-auto=create

Part 2 Advance

  • Multiple table
  • Relationship between tables

One-to-One Mapping

e.g: Instructor -> Instructor Detail

One-to-Many Mapping

e.g:

  • Instructor -> (Many) Course
  • (Many) Course -> Instructor

Many-to-Many Mapping

e.g: (Many) Student -> (Many) Course

  • Primary key: identify a unique row in a table
  • Foreign key:
    • Link tables together
    • a field in one table that refers to primary key in another table

Cascade 級聯

  • Obviously Cascade only exist when table has relationship
  • Apply the same operation to related entities
    • e.g when save a Instructor, it also save its Instructor_detail, same as delete (known as CASCADE DELETE)

However like (Many) Student -> (Many) Course

  • when del a student, should course also be del? of coz not
  • just remove the student from the course

so dev need to conf cascading based on the use case

Fetch Types: Eager vs Lazy Loading

  • Eager: retrieve everything
  • Lazy: retrieve on request

Uni-Directional

Instructor -> Instructor_detail

  • gotta load Instructor first

Hands-on One-To-One

Setup 2 tables with a FK

FK Note

Main purpose is to preserve relationship between tables

  • Referential Integrity (引用完整性)
  • Prevents operations that would destroy relationship
  • Ensures only valid data is inserted into the foreign key column
    • Can only contain valid reference to primary key in other table, otherwise error will be thrown
  • By default, no ops are cascaded
