02 - Web Crawler

Understand the problem

Purpose: index the web for search engines / collect data for research / monitor websites for changes

1.1 Functional

  1. Crawl the full web starting from seed urls
  2. Extract text data and store

1.2 Non-Functional

  1. Fault tolerance (failed to fetch page)
  2. Politness (robots.txt)
  3. Scale to 10B pages
  4. Efficient to crawl in 5 days (3, 4 is the target)

2.1 API

Input: Seed URLs

Output: Text data extracted from web pages

2.2 Data flow

  1. Take a URL from frontier and get its IP from DNS
  2. Use IP to fetch HTML
  3. Extract text from HTML
  4. Store text in DB
  5. Extract URL from HTML then add to frontier

3. High Level Design

  • Frontier Queue, Crawler, DNS, Webpage, S3

Follow up: Use what URL as seed URL

  • provided to you?
  • need to come up with them yourself?
    • starting with the most popular search engines, news sites, social media platforms, and/or web directories.

4. Deep Dive (Map to Non-Functional)

Fault tolerance (failed to fetch page)

S: Get IP / Fetch HTML can be failed due to unstable network -> lose progress

  • The server might be down, the connection might be slow, the page might be too large, etc.

T: break down crawler to

  1. URL Fetcher: fetch then store raw HTML in blob + retry failure URL
  2. Text & URL Extraction: …
  • DynamoDB is fine here. PostgreSQL/MySQL

sub1: What about if we fail to fetch a URL?

  1. Bad Solution: In Memory Timer

  2. Good Solution: Kafka with Manual Exponential Backoff

  3. Great Solution: SQS with Exponential Backoff

sub2: What happens if a crawler goes down?

  • rely on frontier queue (Kafka vs SQS…)

Politness and adhere to robots.txt

1. Delay -> Rate limiting

Redis

Scale to 10B pages + crawl them in 5 days Efficiently

Crawler

  • AWS ecosystem, a network optimized instance can handle about 400 Gbps.
  • 400 Gbps / 8 bits/byte / 2MB/page = 25,000 pages/second
  • let say only can use 30% of it: 25,000 pages/second * 30% = 7,500 pages/second.
  • 10,000,000,000 pages / 7,500 pages/second = 1,333,333 seconds = 15.4 days for a single machine.
  • finish in 5 days: 15.4 days / 4 machines = 3.85 days

Parser

  • scale dynamically based on the number of pages in the Parsing Queue -> can be done by Lambda functions, ECS tasks, or any other serverless technology.

DNS

  1. Unlimit money -> buy better rate limit of DNS
  2. DNS caching -> redis
  3. Use Multiple DNS providers

Page alr be crawled / same content from 2 URL

  1. Great Solution: Hash and Store in Metadata DB w/ Index

  2. Great Solution: Bloom Filter (might be Overkill)

Extra

  1. How to handle dynamic content: Many websites are built with JavaScript frameworks like React or Angular. This means that the content of the page is not in the HTML that is returned by the server, but is instead loaded dynamically by the JavaScript. To handle this, we’ll need to use a headless browser like Puppeteer to render the page and extract the content.

  2. How to monitor the health of the system: We’ll want to monitor the health of the system to ensure that everything is running smoothly. We can use a monitoring service like Datadog or New Relic to monitor the performance of the crawlers and parser workers and to alert us if anything goes wrong.

  3. How to handle large files: Some websites have very large files that we may not want to download. We can use the Content-Length header to determine the size of the file before downloading it and skip files that are too large.

  4. How to handle continual updates: While our requirements are for a one-time crawl, we may want to consider how we would handle continual updates to the data. This could be that we plan to re-train the model every month or that our crawler is for a search engine that needs to be updated regularly. I’d suggest adding a new component “URL Scheduler” that is responsible for scheduling URLs to be crawled. So rather than putting URLs on the queue directly from the parser workers, the parser workers would put URLs in the Metadata DB and the URL Scheduler would be responsible for scheduling URLs to be crawled by using some logic base on last crawl time, popularity, etc.
